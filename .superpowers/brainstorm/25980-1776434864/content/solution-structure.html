How should the solution be structured?

This affects how modules are isolated, how dependencies are enforced, and how easy it is to split into microservices later.

A

Single Project + Module Folders ⭐ Recommended

One Gruuber.Api project. Each module is a folder (Modules/Ride/, Modules/Order/, etc.) with its own Commands, Queries, and Domain sub-folders. Shared kernel is a folder too.

Pros

  • Simple to start, zero friction
  • EF Core migrations straightforward
  • Easy DI wiring

Cons

  • Module boundaries enforced by convention only
  • Harder to split later without refactor
B

Multi-Project Solution

Separate class library projects per module: Gruuber.Rides, Gruuber.Orders, Gruuber.Payments, etc. A thin Gruuber.Api host wires everything together. Shared kernel is Gruuber.SharedKernel.

Pros

  • Compiler-enforced boundaries
  • Each module can be extracted to a microservice with minimal refactor
  • Clear dependency graph

Cons

  • More overhead up front (project refs, migrations setup)
  • More files and solutions to manage
C

Hybrid — Shared Kernel Library + Flat Modules

Gruuber.SharedKernel is a separate project (interfaces, base classes, domain primitives). All modules live as folders inside Gruuber.Api. Best of both: enforced kernel contract, simpler module setup.

Pros

  • Kernel contract is explicit and versioned
  • Modules stay simple
  • Migration path to multi-project is incremental

Cons

  • Module boundaries still convention-only
  • Slightly more setup than option A
