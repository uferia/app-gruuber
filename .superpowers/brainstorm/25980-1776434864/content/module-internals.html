How should each module be organized internally?

This is the folder/namespace structure inside each project like Gruuber.Rides.

A

Clean Architecture Layers ⭐ Recommended

Classic layered split inside each module. Enforces that Domain has zero dependencies.

Gruuber.Rides/
📁 Domain/  — Ride entity, IRideRepository, domain events
📁 Application/  — Commands/, Queries/, IOutboxPublisher usage
📁 Infrastructure/  — EF Core, Kafka consumer, Redis
📁 Contracts/  — DTOs, request/response models
📄 RidesModule.cs  — IServiceCollection extension
B

Vertical Slice (Feature Folders)

Each use-case is a self-contained slice. No shared layers — a slice owns its own handler, validator, and DB access.

Gruuber.Rides/
📁 RequestRide/  — Command, Handler, Validator, Response
📁 MatchDriver/  — Command, Handler, Scoring logic
📁 GetRideStatus/  — Query, Handler, RideView projection
📁 Shared/  — Ride entity, IRideRepository
📄 RidesModule.cs
C

Flat CQRS Split

Simple top-level Commands/Queries split. No strict layering — good for smaller modules like Payments or Tracking.

Gruuber.Rides/
📁 Commands/  — RequestRideCommand, MatchDriverCommand
📁 Queries/  — GetRideStatusQuery
📁 Domain/  — Ride, RideStatus, IRideRepository
📁 Infrastructure/  — EF Core, Kafka, Redis
📄 RidesModule.cs
