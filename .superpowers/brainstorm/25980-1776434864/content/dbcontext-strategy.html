How should DbContext be scoped?

This affects how modules share or isolate their database access — and how easy schema changes are to manage.

A

One DbContext per Module ⭐ Recommended

Each module owns its own DbContext and its own EF Core migrations. Modules never reference each other's DbContext.

Contexts
Gruuber.Rides → RidesDbContext (rides, ride_views, outbox)
Gruuber.Orders → OrdersDbContext (orders, order_items, outbox)
Gruuber.Payments → PaymentsDbContext (payments, outbox)
Gruuber.Auth → AuthDbContext (users, refresh_tokens)

Pros

  • True module isolation
  • Migrations per module — safe deploys
  • Microservices split is trivial

Cons

  • Cross-module joins require Kafka or API calls
  • More migration projects to manage
B

Single Shared DbContext

One AppDbContext in Gruuber.Infrastructure owns all tables and migrations.

Gruuber.Infrastructure
AppDbContext → rides, orders, payments,
               order_items, outbox, users...

Pros

  • Simple — one migration history
  • Cross-entity queries are easy

Cons

  • Breaks module isolation
  • All modules depend on Infrastructure
  • Hard to split later
