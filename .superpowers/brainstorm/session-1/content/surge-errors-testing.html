Surge Pricing Design — Section 3: Error Handling, Observability & Testing

Failure modes, logging, metrics, and test strategy.

Error Handling

Scenario Handling
Redis cache miss (cold start or eviction) Fall through to DB query, populate cache — ride creation not blocked
Redis unavailable Fall back to direct DB query — log Warning; no error surfaced to rider
No surge_config for region/type Default multiplier = 1.0 (no surge) — ride proceeds normally
Multiplier exceeds max_multiplier Clamped to max_multiplier — never surfaces raw tier value to rider
available_drivers = 0 Ratio = active_requests / 1 (avoid divide-by-zero); highest tier likely triggers
Admin updates config mid-high-demand Cache invalidated immediately; in-flight rides keep their locked fare unchanged

Observability

Log fields
 TraceId, SpanId, RegionId, RideId/OrderId, SurgeMultiplier, SurgeReason

Metric Purpose
surge_active_{regionId}_{rideType} Current multiplier gauge per region/type
surge_cache_misses Frequency of Redis fallback to DB
surge_rides_pct % of rides/orders booked at multiplier > 1.0
surge_calc_duration_ms P95 target < 10ms (must stay within request SLA)

Testing

Unit Tests (Moq)
 · Demand ratio below all thresholds → multiplier = 1.0, no surge fields in response
 · Demand ratio hits tier → correct multiplier returned
 · Multiplier exceeds max_multiplier → clamped correctly
 · Active time rule → time rule multiplier used, demand ratio skipped
 · Time rule inactive (outside window) → demand ratio evaluated
 · Redis unavailable → falls back to DB without error
 · No surge_config for region → multiplier defaults to 1.0

Integration Tests (Testcontainers — Postgres + Redis)
 · Book ride during active time rule → final_fare = base × rule multiplier, locked in DB
 · Book ride at high demand → correct tier multiplier applied and persisted
 · Admin updates config → Redis invalidated → next request uses new config
 · GET /v1/surge/estimate returns correct multiplier matching subsequent booking
 · available_drivers = 0 → no divide-by-zero, highest tier triggers

Fare Lock Invariant
 · Assert: ride.final_fare never changes after initial INSERT regardless of subsequent config updates
