Ride Pooling Design — Section 2: Matching Flow & State Machine

How a pool ride request moves from creation to driver dispatch.

Pool Ride Request Flow

POST /v1/rides/request { ride_type: "pool" }
1. Validate pool fare eligibility (pool_region_rates exists for region)
2. Create ride row (status=pool_queued, ride_type=pool, pool_slot=NULL)
3. Write ride_pool_queued event to outbox
4. Push rider into Redis pool_queue:{regionId} sorted set (score=timestamp)
5. Return 202 Accepted { status: "pool_queued", match_timeout_secs }

Pool Matcher (Background Service per region)

PoolMatcherService — runs on Kafka consumer: ride_pool_queued
On ride_pool_queued event:
  1. Scan pool_queue:{regionId} for oldest waiting rider
  2. Compute route detour — reject if detour > max_detour_km
  3. If compatible match found:
    a. Atomically remove both from Redis queue (Lua script)
    b. Generate pool_trip_id (UUID)
    c. UPDATE both rides: pool_slot=1/2, pool_trip_id=X, status=matched
    d. Run driver matching score (existing algorithm, pool-capable drivers only)
    e. Write ride_pool_matched to outbox for both ride IDs
  4. If no match: leave in queue (TTL handles timeout expiry)

Timeout Handling (Redis TTL Expiry)

On pool_queue TTL expiry:
  1. Scheduled job sweeps expired entries (every 30s)
  2. Notify rider via SignalR: "No pool match found — accept solo fare?"
  3. If rider accepts: convert ride to solo, re-enter normal matching
  4. If rider declines / no response in 60s: cancel ride, status=cancelled

Extended Ride State Machine

Solo (unchanged):
requested → matched → en_route → arrived → completed → cancelled

Pool (new states in purple):
pool_queued → pool_matched → matched → en_route → partial_dropoff → arrived → completed → cancelled

partial_dropoff = Rider 1 has been dropped off, driver en route to Rider 2's destination
