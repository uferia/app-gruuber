Ride Pooling Design — Section 3: API, Events & Error Handling

Endpoints, Kafka event contracts, and failure scenarios.

API Changes

Existing endpoint — extended, not modified
POST /v1/rides/request
Body (new optional field):
  { origin, destination, ride_type: "solo" | "pool" }
Response (pool):
  202 Accepted { ride_id, status: "pool_queued", match_timeout_secs, discounted_fare_estimate }

GET /v1/rides/{id}  — unchanged, returns pool_slot + pool_trip_id if pool ride

POST /v1/rides/{id}/accept-solo-upgrade  — new: rider consent to fallback
  202 Accepted { status: "requested" } (re-enters solo matching queue)

Kafka Event Contracts

ride_pool_queued → { ride_id, rider_id, region_id, origin, destination, requested_at }
ride_pool_matched → { pool_trip_id, ride_ids:[id1,id2], driver_id, region_id, pickup_order }
ride_pool_timeout → { ride_id, rider_id, region_id, reason: "no_match", notify_user: true }
ride_pool_upgraded → { ride_id, rider_id, region_id, previous_status: "pool_queued" }
ride_partial_dropoff → { pool_trip_id, completed_ride_id, remaining_ride_id, region_id }

All existing ride_* events (ride_matched, ride_completed, etc.) fire per-ride as before

Error Handling & Edge Cases

Scenario Handling
Double-match race (two matchers pick same rider) Lua atomic remove from Redis queue; loser retries with next candidate
No pool-capable drivers in region Return 202 with pool_queued; fallback prompt after timeout
Rider 2 cancels after pool_matched Rider 1 continues as solo; driver re-matched if fare delta significant; compensation event emitted
Driver cancels mid pool trip Both rides enter re-matching; both riders notified via SignalR
No pool_region_rates config for region 400 Bad Request — pool not available in this region
