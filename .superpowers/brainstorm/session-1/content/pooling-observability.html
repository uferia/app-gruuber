Ride Pooling Design — Section 4: Observability & Testing

Logging, metrics, and test strategy for pool rides.

Structured Logging

All pool log entries include:
  TraceId, SpanId, RegionId, RideId, PoolTripId, PoolSlot

Info  → pool_queued, pool_matched, partial_dropoff, pool_completed
Warning → pool_timeout (no match found), solo upgrade accepted/declined
Error  → Lua atomic failure, matcher consumer DLQ, double-match collision

Key Metrics to Monitor

Metric Purpose
pool_queue_depth_{regionId} Alert if queue backs up — signals low driver supply
pool_match_rate % of pool requests successfully matched
pool_timeout_rate % falling back to solo or cancelling
pool_match_latency_ms Time from pool_queued → pool_matched
pool_detour_km Histogram of actual detour added per trip

Test Strategy

Unit Tests (Moq)
 · PoolMatcherService: compatible match found, detour exceeded, no candidates
 · Fare calculation: discount applied from pool_region_rates
 · State machine: valid + invalid transitions (e.g. pool_queued → completed is illegal)

Integration Tests (Testcontainers)
 · Full pool match: 2 riders request → both matched → driver assigned
 · Timeout flow: 1 rider queued, TTL expires → SignalR nudge → solo upgrade
 · Race condition: 2 matchers attempt same pair simultaneously → only one succeeds
 · Rider 2 cancels after match → Rider 1 continues solo

Privacy Guard
 · Assert GET /v1/rides/{id} for Rider A never returns Rider B's pickup/dropoff coords
