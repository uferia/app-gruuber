Surge Pricing — Architecture Approaches

All 3 approaches compute an upfront locked fare with surge baked in. The difference is where and how the multiplier is evaluated.

A

⚡ Inline Fare Calculator with Redis Cache Recommended

A SurgePricingService is called inline at ride/order request time. It reads surge config from a Redis-cached surge_config key (TTL 60s, populated from DB). It computes the active multiplier from demand/supply ratio + time-of-day rules, locks the fare, and returns it. The multiplier and locked fare are persisted on the ride/order row.

Pros

  • Simple — no new background workers
  • Fare locked at request time (price guarantee)
  • Redis cache keeps latency low (no DB hit per request)
  • Fits existing request flow cleanly

Cons

  • Demand/supply ratio computed per request (lightweight but not zero cost)
B

🔄 Pre-computed Surge Level per Region

A background worker runs every 30s and writes the current surge multiplier per region to Redis (surge_level:{regionId}). Ride/order request handlers simply read this value — no computation at request time. Fare is locked from this cached value.

Pros

  • Zero computation overhead at request time
  • Consistent multiplier across all requests within the same 30s window

Cons

  • New background worker to maintain
  • Surge can be up to 30s stale at time of booking
  • More moving parts
C

🏗 Dedicated Surge Module (Kafka-driven)

A standalone Gruuber.Surge module subscribes to ride_requested and ride_completed events to maintain demand/supply counters. It publishes surge_level_changed events when the multiplier crosses a tier boundary. Ride handlers subscribe to the current level via Redis.

Pros

  • Most event-driven — fits the architecture vision
  • Clean domain separation

Cons

  • Highest complexity — new module, new events, new consumers
  • Event lag could cause stale surge level at booking
  • Over-engineered for stepped-tier multiplier logic
