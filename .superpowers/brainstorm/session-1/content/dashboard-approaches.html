Driver & Restaurant Dashboards — Architecture Approaches

All 3 approaches use a dedicated Gruuber.Analytics module with pre-aggregated Kafka-driven read models. The difference is in how aggregation snapshots are structured.

A

📅 Time-Bucketed Snapshot Tables Recommended

Three dedicated tables: driver_stats_daily, restaurant_stats_daily, admin_stats_daily — one row per entity per day. Kafka consumers upsert into these on every ride_completed / order_delivered event. Weekly and monthly views are derived at query time by summing daily rows. Export jobs read directly from these tables.

Pros

  • Fits CQRS perfectly — Kafka consumers own writes
  • Simple queries (SUM over date range)
  • Easy to backfill on schema change
  • No extra infrastructure

Cons

  • Monthly summaries require summing up to 31 rows
  • Menu-item-level stats need a 4th table
B

🔄 PostgreSQL Materialized Views

Raw events land in an analytics_events table via Kafka consumers. PostgreSQL materialized views (REFRESH MATERIALIZED VIEW CONCURRENTLY) compute daily/weekly/monthly summaries on a schedule (e.g. every 5 min). Dashboard queries hit the materialized views directly.

Pros

  • Flexible — change view logic without Kafka consumer changes
  • Rich SQL aggregations available

Cons

  • Refresh lag (up to 5 min stale data)
  • Materialized views can lock under high write load
  • More complex ops
C

⚡ Hybrid: Daily Snapshots + Rolling Cache

Same daily snapshot tables as Option A, but a Redis cache layer stores the current-day running totals (updated in real-time per event). Dashboard queries merge Redis (today) + Postgres (historical days). Gives near-real-time accuracy for the current day.

Pros

  • Most responsive — today's data is always fresh
  • Historical data stays in Postgres

Cons

  • Redis + Postgres merge logic adds complexity
  • Redis loss means today's totals reset (until Kafka replay)
  • Overkill for daily/weekly/monthly-only requirement
