Surge Pricing Design — Section 2: Fare Calculation Flow & API

How SurgePricingService computes the multiplier and locks the fare at booking time.

SurgePricingService — Multiplier Resolution

Called inline during POST /v1/rides/request and POST /v1/orders/create
1. Load config from Redis surge_config:{regionId}:{rideType} (TTL 60s)
   If cache miss → query DB, populate cache

2. Check surge_time_rules — is current time within an active rule?
   If YES → multiplier = time_rule.multiplier, reason = 'time_rule'
   If NO → evaluate demand/supply ratio:

3. Demand/supply ratio:
   active_requests = count of rides/orders in 'requested' status for region
   available_drivers = ZCARD driver_locations:{regionId} (Redis GEO)
   ratio = active_requests / max(available_drivers, 1)

4. Walk surge_config tiers (desc by threshold) → pick highest matching tier
   multiplier = min(tier.multiplier, max_multiplier), reason = 'demand'
   If no tier matches → multiplier = 1.0, reason = null

5. Return: { multiplier, reason, base_fare, final_fare = base_fare * multiplier }

Fare Lock on Ride/Order Creation

In same DB transaction as ride/order INSERT:
  rides.base_fare = calculated base fare
  rides.surge_multiplier = resolved multiplier
  rides.final_fare = base_fare × surge_multiplier
  rides.surge_reason = 'demand' | 'time_rule' | null

final_fare never changes after this point — rider is price-guaranteed

API Response Changes (non-breaking)

POST /v1/rides/request → 202 Accepted
  { ride_id, status, fare_estimate: { base_fare, final_fare, surge_multiplier?, surge_reason? } }
  surge_multiplier + surge_reason only included if multiplier > 1.0

POST /v1/orders/create → 202 Accepted
  { order_id, status, fare_estimate: { base_fare, final_fare, surge_multiplier?, surge_reason? } }

GET /v1/rides/{id} — returns base_fare, final_fare, surge_multiplier, surge_reason on the ride

GET /v1/surge/estimate?region_id=&ride_type=ride|food — new: preview fare multiplier before booking
  { surge_multiplier, surge_reason, valid_for_secs: 30 }

Admin Config Invalidation

PUT /v1/admin/surge/config  (admin role only)
  → Updates surge_config / surge_time_rules in DB
  → Immediately deletes Redis key surge_config:{regionId}:{rideType}
  → Next request repopulates cache from fresh DB data
