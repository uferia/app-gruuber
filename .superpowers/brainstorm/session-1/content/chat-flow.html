In-App Chat Design — Section 2: Thread Lifecycle & SignalR Flow

How threads are created, how messages flow via ChatHub, and how threads close.

Thread Creation (Kafka-driven)

ChatThreadConsumer — listens to ride/order events
On ride_matched → Create thread (context=rideId, channel='rider_driver')
                 Add participants: rider (display="Rider"), driver (display="Driver")

On order_accepted → Create thread (context=orderId, channel='customer_restaurant')
                   Create thread (context=orderId, channel='driver_restaurant')
                   Add participants per channel

On ride_completed / order_delivered → Set thread.closes_at = now() + 6h
                                   Emit chat_thread_closing via SignalR to participants

Scheduled sweep job (every 5 min): mark threads past closes_at as 'read_only'

ChatHub (SignalR) — Message Flow

Gruuber.Chat owns a dedicated ChatHub — separate from TrackingHub
Client connects: JoinThread(threadId)
  → Validate JWT: user must be in chat_participants for this thread
  → Add to SignalR group: chat:{threadId}
  → Mark any unread messages as 'delivered'

Client sends message: SendMessage(threadId, content, messageType)
  → Validate thread is 'active' (reject if 'read_only' or 'closed')
  → Persist to chat_messages (status='sent')
  → Broadcast to group chat:{threadId}: ReceiveMessage(message)
  → Update status='delivered' for online recipients

Recipient opens message: MarkRead(messageId)
  → Update chat_messages.status='read', read_at=now()
  → Notify sender via SignalR: MessageRead(messageId)
  → Single tick = 'sent', Double tick = 'read'

Redis backplane: fan-out works across all Gruuber instances
(same Redis already used for TrackingHub backplane)

REST API (for history + quick replies)

GET /v1/chat/threads?context_id={rideOrOrderId}
  → Returns thread(s) the caller participates in for this context

GET /v1/chat/threads/{threadId}/messages?before=&limit=20
  → Paginated history (cursor-based, sorted by sent_at desc)

GET /v1/chat/quick-replies?role=driver|rider|restaurant
  → Returns active templates filtered by caller's role
