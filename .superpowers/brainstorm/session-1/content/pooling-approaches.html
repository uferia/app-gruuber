Ride Pooling — Architecture Approaches

3 approaches for implementing live 2-rider pool matching. Pick your preferred direction.

A

🏷 Ride Type Variant Recommended

Add a ride_type (solo | pool) and pool_slot (1 | 2) to the existing rides table. A background pool matcher (Kafka consumer) holds a Redis sorted-set queue per region. When two riders are paired, both ride records are linked via a pool_trip_id FK. The driver's matching score already exists — pool matching reuses it, filtered to pool-mode drivers with an open slot.

Pros

  • Minimal schema change
  • Reuses existing matching + outbox flows
  • Low blast radius on existing modules

Cons

  • Ride table grows wider
  • Slightly more complex queries for pool state
B

🗂 Dedicated Pool Trip Entity

Introduce a pool_trips table as the aggregate root (status, driver_id, region_id, version). Individual rides rows are children referencing pool_trip_id. All state transitions happen at the pool_trip level, emitting pool_trip.* Kafka events. Pool and solo rides are fully separated.

Pros

  • Clean domain separation
  • Pool lifecycle owned by one aggregate

Cons

  • More tables + new event contracts
  • Read model needs updating for pool trips
  • Higher implementation cost
C

⏱ Redis Pool Queue Service

A standalone pool-matching service (could be a hosted service in the monolith) manages a Redis Sorted Set as a waiting queue per region. On each new pool request, it scans for a compatible waiting rider within geo-proximity threshold. If found, pairs them and dispatches a driver. No schema changes to rides — pairs are resolved before a ride is written.

Pros

  • Very fast matching (all in Redis)
  • Decoupled from DB until pair confirmed

Cons

  • Redis loss = lost queue (no durability)
  • Complex recovery on failure
  • Harder to audit match history
